# 免费域名查询实现调研

调研日期：2026-06-11

## 结论先说

如果目标是“免费 API、不使用 GoDaddy 这类付费额度”，可行路线不是完全没有接口，而是使用公开协议：

- RDAP：现代注册局查询协议，HTTP/JSON，适合做第一查询源。
- WHOIS：传统 43 端口查询，格式不统一，但很多 ccTLD 仍然必须依赖它。
- IANA bootstrap：用来发现不同后缀对应的 RDAP/WHOIS 服务。
- DNS：只能作为“已被使用/有解析”的辅助证据，不能证明域名可注册。

真正不能免费保证的是“溢价价格/最终结算价”。RDAP/WHOIS 能帮助判断注册状态，但不会可靠返回注册商最终售价、溢价价、税费、首年优惠、资格限制。这个部分只有注册商接口、注册商后台或下单页能最终确认。

所以最适合当前项目的方向是：

> 免费模式继续做，但策略改成“强保守”：只有 RDAP/WHOIS 给出明确可注册证据时才显示“可注册”；任何超时、空响应、冲突、DNS 只能辅助的结果，都进入“需复核”，不要进入可注册列表或可能溢价列表。

## 逐项分析

### 1. Toolshu

公开页面写的是“通过 WHOIS 接口实时查询域名注册状态”，并且说明网络异常或接口超时可能导致失败。它还限制普通用户每次最多 200 个，VIP 2000 个。

可参考点：

- 产品形态：批量输入、生成、分类、导出。
- 状态模型：未注册、已注册、失败。

不建议直接照搬：

- 没有公开源码，无法确认它是否会把空 WHOIS、超时、被限流误判成未注册。
- 它有 VIP 额度，不符合我们“完全免费给客户稳定用”的目标。

### 2. 纳网批量查询

纳网本身就是域名服务商，页面提供批量注册/批量查询，支持查询线程、查询延迟、失败重试。它的准确性优势来自“注册商/销售链路”，不是公开免费协议。

可参考点：

- 线程数、延迟、失败重试应该给后台可配置。
- 查询失败单独分组，用户可以重新查询。

不能依赖点：

- 它不是开放免费 API。
- 注册商内部库存/价格能力，我们无法免费复用。

### 3. 秋玉米

秋玉米主要是批量生成候选域名并查询，支持多后缀、字典组合、双拼三拼等。公开页面没有说明底层协议细节，只能确认它提供 WHOIS 查询入口和批量生成能力。

可参考点：

- 域名生成能力、字典组合、拼音组合。
- 结果导出和候选筛选。

不建议依赖：

- 没有稳定公开 API。
- 不能验证其准确性策略。

### 4. Instant Domain Search

它是商业级域名搜索产品，并且页面说明自己是 ICANN-accredited registrar，支持 800+ 后缀、实时价格、8 个注册商比价、后市场聚合。

这说明一件事：要做到“价格、溢价、下单可用性”非常准，最终绕不开注册商/交易渠道。它不是纯 RDAP/WHOIS 能实现的。

可参考点：

- 用户体验：实时搜索、注册商价格对比、后市场信息。
- 商业模式：免费查询 + 注册跳转/佣金。

不适合当前免费约束：

- 8 个注册商实时价格、后市场聚合，不是免费公开协议能稳定得到的。
- 我们不能承诺相同的价格准确性。

### 5. PlayerYK/dmcheck

这个项目非常值得参考。README 明确写了：

- 免费域名可用性检查，底层是 WHOIS + RDAP。
- 直接查询注册局 WHOIS 43 端口，RDAP 通过 IANA bootstrap fallback。
- 预置 83 个 WHOIS 后缀，RDAP fallback 支持 1000+ 后缀。
- 默认可以作为纯域名可用性检查器运行，不需要注册商数据。
- 注册商价格/跳转是可选模块，且默认关闭。
- 可注册结果默认不缓存，已注册/保留结果可长缓存。

这和我们当前方向高度一致。

值得借鉴：

- 查询状态和注册商价格彻底分离。
- `unknown` 是一等状态，而不是失败后硬判可注册。
- 可注册结果默认不缓存，降低“刚被注册但还显示可注册”的风险。
- 价格来源只做“参考价”，最终价格交给注册商确认。

### 6. PowerWhois

PowerWhois 是功能更重的 Go Web 工具。它支持：

- 批量域名查询。
- 原始 WHOIS 信息弹窗。
- WHOIS、WHOIS + Proxy、DNS、Mix 模式。
- 自定义 WHOIS API。
- 自定义注册 API。
- 线程、失败重试、API 模式。

它自己也说明：部分后缀只能使用 DNS 模式，少数注册商保留域名可能不准确。

可参考点：

- 后台配置：线程、超时、失败重试、不同后缀使用不同查询策略。
- Mix 模式：不同后缀单独配置查询方法。
- 代理：对某些 WHOIS 不稳定地区可能有帮助。

需要谨慎：

- DNS 模式不能作为最终“可注册”证据。
- 自定义注册 API 往往意味着接入第三方或自有注册商接口，不符合“免费公开源”默认目标。

### 7. domain-scanner

这个项目用多种信号判断：

- DNS NS/A/MX。
- WHOIS。
- SSL 证书。
- WHOIS 失败自动重试三次。

源码里一个很好的思想是：服务错误、超时、限流不能当作可注册；没有明确可注册信号时，宁可保守地认为不可注册/不输出为可注册。

可参考点：

- 多信号用于证明“已注册/被使用”很有价值。
- WHOIS 显式可注册文案才能返回可注册。
- 服务错误不进入可注册。

需要注意：

- DNS/SSL 只能证明域名被使用，不能证明没 DNS/没证书就是可注册。
- 它为了降低误报，可能牺牲召回率：有些可注册域名会被漏掉，但这比误报可注册更适合我们的场景。

### 8. saidutt46/domain-check

这个项目是 Rust 库/CLI/MCP，路线非常清晰：

- RDAP-first。
- WHOIS fallback。
- IANA bootstrap。
- 1200+ TLD。
- 并发查询。
- 结果支持 `available=true/false/null`，其中 null 就是未知。

它还有一个关键设计：RDAP 404 不一定可信，有些注册局可能对已注册域名也返回 404，所以它会用 WHOIS corroboration 或 UNKNOWN 处理。

这点正好对应我们项目之前的事故：不能把“查不到/空响应/404”直接当可注册。

可参考点：

- `available: null` 这样的三态模型。
- RDAP 404 需要 WHOIS 佐证。
- 网络和注册局依赖导致 UNKNOWN 是正常状态。

### 9. dotchk

dotchk 是速度型工具。它查 NS 记录，不查 WHOIS。它自己明确说明：

- 查的是 NS records，不是 WHOIS。
- 如果域名已注册但没有配置 nameserver，会误报为 available。
- 适合 discovery 和 bulk scanning，购买前必须 WHOIS 验证。

结论：

- 可以作为“超高速预筛”参考。
- 不能作为我们前台“可注册”最终依据。
- 如果引入，只能把结果标为“DNS 未发现解析，需 WHOIS 复核”，不能标为可注册。

### 10. TLDHunt

TLDHunt 是 Bash 脚本：

- 依赖系统 `whois`。
- 用 `curl` 从 IANA 更新 TLD 列表。
- 通过 grep `Name Server`、`nserver`、`nameservers`、`status: active` 来判断已注册。

优点是极简、免费。

缺点也很明显：

- WHOIS 格式不统一，grep 少数字段会漏判。
- 没有 RDAP。
- 没有严谨的 UNKNOWN/错误状态。

适合学习“轻量 CLI”，不适合作为当前项目准确性核心。

## 是否必须用付费 API？

不必须，至少“域名是否已注册/是否可能可注册”不必须。

免费路线可以做到：

- 查注册状态：RDAP + WHOIS。
- 查注册局权威状态：多数 gTLD 通过 RDAP，部分 ccTLD 通过 WHOIS。
- 批量筛选：可以。
- 高置信“已注册”：可以做到比较可靠。
- 高置信“可注册”：可以做到，但必须保守，失败就不要报可注册。

免费路线不能保证：

- 溢价域名准确价格。
- 注册商最终结算价格。
- 首年优惠、续费价、税费、资格限制。
- 某个注册商是否接受下单。
- 注册局/注册商实时锁定、刚被注册、刚释放等竞态。

## “准确性”怎么提升

目标不是追求“结果越多越好”，而是追求“可注册列表里尽量不出现假阳性”。也就是说：

> 宁可漏掉一些可注册，也不要把已注册域名显示成可注册。

具体策略：

1. RDAP-first，WHOIS fallback。
2. TLD 策略表：`.com/.net`、`.org`、`.cn`、`.io`、`.ai` 等高频后缀分别配置规则。
3. `.cn` 这类不稳定后缀必须双重确认：至少两次明确 `No matching record` 才可注册；任何 `Domain Name:`、空响应、连接重置、超时都不能判可注册。
4. DNS 只作为已注册辅助证据：有 NS/A/MX/SSL，可以提高“已注册/被使用”置信度；没有 DNS 不能说明可注册。
5. RDAP 404 不能无脑当可注册：最好再走 WHOIS；如果 WHOIS 不通，给“需复核”。
6. 可注册结果不长缓存，最多短缓存；已注册结果可以缓存更久。
7. 导出前强制二次复核，只导出二次复核仍明确可注册的域名。
8. UI 上把“不确定”从“可注册/可能溢价”里移出去，单独叫“需复核”。
9. “可能溢价”改名为“高价值词形/价格需复核”，不要暗示已经知道真实溢价。
10. 对客户明确标注：最终下单前以注册商结算页为准。

## 当前项目应该怎么做

建议采用三层结果模型：

### A. 可注册

进入条件必须很严格：

- RDAP authoritative 404，并且对应 TLD 规则允许信任；或
- WHOIS 返回明确未注册文案；或
- 对高风险 TLD，RDAP/WHOIS 两次一致确认。

### B. 已注册

进入条件：

- RDAP 200 并返回注册对象。
- WHOIS 出现 `Domain Name:`、注册日期、注册商、NS、状态码等。
- DNS/SSL 作为辅助证据，但最好不单独作为唯一证据，除非该 TLD 配置为 DNS 预筛。

### C. 需复核

进入条件：

- 超时。
- 空响应。
- 连接重置。
- RDAP 404 但 WHOIS 无法确认。
- RDAP/WHOIS 冲突。
- WHOIS 返回服务错误、限流、被拒绝。
- DNS 无记录但没有注册局明确可注册证据。

这会减少“可注册列表”数量，但会大幅降低用户点进去发现已注册的崩溃感。

## 我的建议

不建议接 GoDaddy API 作为核心方案。它额度少、依赖商业账号，也不能解决所有后缀和所有注册商最终价问题。

建议继续做免费公开源，但产品定位要改成：

> 免费批量候选筛选工具，提供高置信注册状态；对不确定结果强制复核；最终购买前跳转注册商确认。

技术路线：

1. 以 saidutt46/domain-check 和 dmcheck 为主要参考：RDAP-first + WHOIS fallback + UNKNOWN。
2. 借鉴 domain-scanner 的保守策略：服务错误绝不当可注册。
3. 不采用 dotchk/TLDHunt 作为最终判断，只可作为预筛或学习对象。
4. 对 `.cn`、`.com.cn`、`.net.cn` 继续特殊处理，严格双确认。
5. 对“可能溢价”降级为“词形高价值，价格需复核”。

这样这个工具才有使用理由：它不是承诺 100% 替代注册商，而是帮你从几百/几千个候选里快速筛出少量值得手动确认的名字，并尽量不把明显不确定的东西放进“可注册”结果里。
