## 二、已实现的改进

- 三层缓存：硬编码(32) → 进程缓存(24h) → 远程拉取
- 负缓存 + 冷却时间（失败后 5 分钟不重试）
- 已缓存到本地 `iana_bootstrap_cache.json`

### 2.2 置信度评分系统 ✅ `api/check.py`

| 条件 | 置信度 |
|---|---|
| 已注册 + 多源确认 | VERY_HIGH |

| 可注册 + 多源 | VERY_HIGH |

| 可注册 + whois 单源 | MEDIUM |

## 三、已存在但未解决的问题

### 问题 1：DNS 预筛漏斗 ❌ 未实现

**问题**：每个域名都走 HTTP RDAP 请求，对于大量已注册域名浪费严重。

| 指标 | 当前 | 加 DNS 预筛后 |
|---|---|---|
| 已注册域名判定速度 | 500ms-2s (HTTP) | **5ms** (本地 DNS) |
| RDAP 请求量 | 100% | **~20%** (只查无 DNS 的) |
| 被限速概率 | 高 | **低** |

**优先级**：P0（核心加速）

---

### 问题 2：指数退避重试 ❌ 未实现

**问题**：遇到 429/503 等限速直接报错，需要用户手动重查。

**修复方向**：
```
遇到 429 → 等 1.5s 重试 → 等 3s 重试 → 等 6s 重试 → 3次失败才报错
```

**优先级**：P0（无人值守必备）

---

### 问题 3：Vercel 10s 超时限制（已知限制）

**问题**：Vercel Hobby 计划 Serverless Function 最长 10s。

**当前妥协**：单次最多查 20 个域名。

**解决方案**：
- 保持此限制（免费计划就这样）
- 前端做分批提交（每批 20 个，自动排队）
- 或使用 `domain_server.py` 自托管（无超时限制）

**优先级**：P1（用户体验优化）

---

### 问题 4：无 SSE 流式结果 ❌

**问题**：前端必须等所有域名查完才能看到结果。

**当前状态**：`api/check.py` 是一次性 POST 返回。

**修复方向**：
```python
# 在 domain_server.py 或 Render 后端中加 SSE
GET /api/check/stream?domains=...
→ data: {"domain":"google.com", "status":"taken"}
→ data: {"domain":"aiabc.com", "status":"available"}
→ data: {"type":"done", "total":20}
```

**优先级**：P1

---

### 问题 5：无 SQLite 缓存 ❌

**问题**：重复查询同一域名每次都重新请求 RDAP。

**优先级**：P2

---

### 问题 6：.cn 查询冗余（小问题）

**问题**：`api/check.py` 对 .cn 直接走 whois，但 `domain_server.py` 里 .cn 同时也会走 RDAP（if cn in IANA bootstrap），两边不一致。

**优先级**：P2

---

### 问题 7：公共数据源不可靠（已知外部问题）

**问题**：rdap.org 不稳定、CNNIC SSL 失败、部分 TLD 无 RDAP。

**应对（已部分解决）**：
- IANA Bootstrap 多源投票
- RDAP 失败 → WHOIS 自动降级（saidutt46 模式）
- .cn 走 whois 端口 43
