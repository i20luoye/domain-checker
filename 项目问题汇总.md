### 问题 1：浏览器点击查询全部失败

**时间**：09:22
**症状**：用户截图显示所有域名都标"查询失败"
**根本原因**：
- `rdap.verisign.com` 响应不带 `Access-Control-Allow-Origin` 头
- 浏览器 CORS 策略拦截响应，JS 拿不到数据

**解决**：
- 全部改走 `rdap.org` 代理（支持跨域）
- 并发默认 5
- 加重试 3 次 + 429 退避

### 问题 3：不能识别溢价域名

**时间**：10:35、11:00
**用户反馈**：
- aiet.cn 实际是 ¥23 万一口价，但工具只说"可注册"
- 好不容易找到喜欢的域名，点进阿里云发现是溢价

**问题本质**：
- RDAP 协议**只返回"已注册/未注册"**，不返回价格
- 溢价域名从未被注册但价格离谱
- 启发式规则只能猜，**不能保证准确**

**两次解决尝试**：

**尝试 1（启发式规则）**：
- 规则：常见单词、重复字母、顺序字母、回文、词根
- 5 位以上明显误报（aieo.cn 被误判，实际是普通价）
- **结论：启发式不可靠**

**尝试 2（GoDaddy 批量 API）**：
- `POST /v1/domains/available` 一次最多 500 个
- 返回真实 `price` 和 `currency`（微单位 ÷1,000,000）
- 按 TLD 阈值表判断溢价（com>$50, ai>$500, cn>$200）
- **OTE 沙箱不返完整溢价库**——又遇到限制

**当前结论**：
- 沙箱 Key：能验证"已注册/可注册"判断正确，但**不能识别真溢价**
- 生产 Key：能识别真溢价，**但会真实扣费**（按 API 调用次数，几乎免费，但有风险）
- 用户决定中

### 问题 5：多服务器自动故障切换

**时间**：09:45
**用户要求**：多服务器同时查询，自动切换失败服务器
**实测 RDAP 源 CORS 兼容性**：

| 源 | CORS 支持 | 备注 |
|---|---|---|
| verisign.com | ✅ | .com/.net 权威源 |
| iana.org | ✅ | 任何 TLD 都支持（带 ACAO=*） |
| gname.com | ✅ | |
| publicinterestregistry.org | ✅ | .org |
| rdap.org | ❌ | 返回 302 重定向（之前误以为是抽风） |
| markmonitor | ❌ | |
| cnnic | ❌ | |
| nic.io | ❌ | |

**新架构**：
- 每个后缀配 1-3 个独立源
- .com: verisign(主) + iana + gname
- .net: verisign(主) + iana
- .org: publicinterestregistry(主) + iana
- .cn/.io: rdap.iana.org（CNNIC 不支持 CORS）

**查询策略**：
1. 先打主源（6 秒超时）
2. 主源明确返回 200/404 直接用
3. 主源失败/超时/限速/5xx → 并发打所有源投票
4. 投票规则：任一源 200 = taken；任一源 404 = available；全失败 = error

### 问题 7：用户电脑浏览器 CORS 完全拦截

**时间**：10:16
**用户截图**：35,152 个查询 1 秒内全部失败，0 个可注册
**线索**：
- 402,274 个/分钟（异常高），1 秒完成 35,152 个查询

- 浏览器版工具在这台电脑/网络上**完全不可用**

**解决方案**：
- 改用 Python 脚本，绕过浏览器 CORS 限制
- 重写 `domain_checker.py` 为多源 + 通用配置版
- 实测：6% 进度时已发现 608 个可注册 / 1498 已注册 / 0 错误
- 速度 19-30/秒，比浏览器快 30-100 倍

### 问题 10：.cn RDAP 数据假阳性（可注册但实际已注册）

**时间**：10:50
**用户截图**：所有 .cn 域名都被标"可注册"，但实际阿里云是已注册
**关键诊断**：
- `rdap.iana.org` 对 .cn 返回 404（实际已注册的 google.cn 也 404）—— IANA 没正确路由
- `rdap.cnnic.cn` SSL 失败，IP 解析为 `198.18.1.169`（RFC 5737 文档示例 IP），CNNIC RDAP 实际未配置
- IANA bootstrap 文件 `dns.json` 中根本没有 .cn 注册记录
- **结论：.cn 顶级域在 RDAP 体系下实际不可查**

**最终解决**：
- 用 CNNIC 的 whois 端口 43（标准 whois 协议，可用且稳定）
- 新增 `whois://` URL 协议处理
- 解析规则：
  - 包含 "Domain Name:" = 已注册（200）
  - 包含 "No matching" = 可注册（404）
  - 空响应 = 警告
- 实测 216 个 .cn 域名 7 秒跑完，114 可注册 + 102 已注册，0 错误
- 用户截图里的 aiey.cn/aiev.cn/aifa.cn 实测都已被注册

### 问题 11：搜索框被新结果覆盖

**用户反馈**：搜索某关键词时显示结果，但马上被新查询结果覆盖掉

- 新增 `onSearchInput()` 实时更新提示状态

### 问题 12：溢价检测误报（5位域名）

**时间**：10:35
**用户截图**：aieo.cn / aiei.cn / aiee.cn 都被标"可能溢价"，但实际阿里云是普通价或未列出
**根因**：原规则把"全元音/全辅音/顺序字母"应用于所有长度，5 位以上明显误报
**修复**：
- 分长度细判（4 位/5 位/6+位），每档不同规则
  - ≤3 位：稀缺全溢价
  - 4 位：稀缺 + 模式（顺序/回文/abab/aabb）
  - 5 位：只判词根（去掉前 1-2 字母后是常见单词 = 溢价）
  - 6+ 位：只判 abcabc 重复
- 新增"溢价原因"列：用户能看到具体为什么被标溢价
  - 例：aibox=词根(box), aiaa=全元音, aicba=回文
- 关键：不再"宁可错杀"，5 位以上无确凿模式 = 普通价

## 核心经验教训

1. **RDAP 协议的本质限制**：HTTP 404=可注册、200=已注册，**不包含价格信息**。要做"溢价判断"必须用商业 API。
2. **CORS 是浏览器工具的最大障碍**：跨域访问公共数据源时，浏览器安全策略会拦截。Python 脚本是绕过的最简方式。
3. **多源并发投票是稳定性的关键**：单一 RDAP 源失败率太高（rdap.org 经常抽风），多源投票架构能让"全失败"概率降到 0.1% 以下。
4. **.cn 是 RDAP 体系的黑洞**：CNNIC 自有 RDAP 不支持 CORS+SSL 失败，IANA 也没正确路由。**whois 端口 43 是唯一靠谱的方案**。
5. **沙箱和生产环境的功能差异巨大**：GoDaddy OTE 沙箱不返完整溢价库，测试时容易误以为"一切都正常"。生产环境才会暴露真问题。
6. **启发式规则只能兜底**：模式匹配（短域名/词根/顺序字母）能猜个大概，但**不能保证准确**。要做"用户能直接下单的查询结果"，必须用真实 API 拿价格。
